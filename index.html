Manual Interativo de GitHub para acompanhar uma PAP

Este site foi criado para servir como manual de apoio durante a realização de uma PAP, ajudando a organizar o projeto desde o primeiro dia até à entrega final. Ao longo do trabalho, vais criar ficheiros, alterar versões, melhorar documentos, registar evidências e, por vezes, corrigir erros. Com um repositório bem organizado, fica muito mais simples manter tudo controlado.

A ideia é simples: o repositório GitHub funciona como a “pasta oficial” do projeto. Nele ficam guardados: ficheiros do projeto, documentação (planeamento/relatório), prints/evidências e o registo de alterações ao longo do tempo. Assim, o projeto fica mais fácil de acompanhar, melhorar e justificar com evidências reais.

Para que serve este manual

Objetivo: apoiar o desenvolvimento do projeto do início ao fim, com organização e evidências.

Ao longo do projeto é normal existirem várias versões de ficheiros, mudanças de ideias, correções e melhorias. Sem um método de organização, torna-se fácil perder trabalho, confundir versões ou não conseguir demonstrar aquilo que foi feito em cada etapa. Este manual guia-te nas rotinas essenciais para manteres o projeto “arrumado”.

Em termos práticos, vais usar o GitHub para:

  • Criar e manter um repositório do projeto.
  • Organizar pastas (código, docs, imagens/prints, recursos).
  • Registar evidências do trabalho (prints e documentos).
  • Trabalhar em segurança com branches e Pull Requests.
  • Gerir tarefas com Issues (o que falta fazer, o que já foi feito).
  • Publicar um site do projeto com GitHub Pages (quando fizer sentido).
Sugestão: utiliza este manual como referência semanal. Sempre que avançares uma fase, confirma se o repositório continua limpo, organizado e atualizado (ficheiros no sítio certo e evidências guardadas).

Como usar este manual ao longo do projeto

Uma rotina simples, repetível e eficaz durante toda a PAP.

Este manual não é para “ler uma vez e esquecer”. A ideia é ir usando durante o projeto: no início para preparar o repositório, durante o desenvolvimento para manter o trabalho atualizado e, no final, para garantir que tudo está pronto para entregar e apresentar.

1) No início

Criar base do projeto e organizar estrutura.

Arranque
  • Criar repositório com nome claro e descrição.
  • Definir pastas (src/, docs/, images/).
  • Adicionar uma explicação inicial (objetivo + plano).

2) Todas as semanas

Atualizar trabalho e registar evidências.

Rotina
  • Adicionar ficheiros novos e documentação.
  • Guardar prints/evidências do que foi feito.
  • Fechar Issues concluídas e criar novas tarefas.

3) Em mudanças grandes

Trabalhar com branches e PR.

Segurança
  • Criar branch para alterações relevantes.
  • Abrir Pull Request para integrar na main.
  • Guardar evidências (PR criado e merge feito).

4) Antes da entrega

Revisão final e publicação.

Final
  • Conferir estrutura e ficheiros finais.
  • Confirmar links e evidências no repositório.
  • Ativar/confirmar GitHub Pages (se aplicável).
O que interessa é consistência: projeto organizado + evidências regulares + evolução documentada.

Manual Interativo de GitHub para acompanhar uma PAP

Este site foi criado para servir como manual de apoio durante a realização de uma PAP, ajudando os alunos a manterem o projeto organizado, documentado e com evidências claras do trabalho desenvolvido ao longo do tempo.

A ideia é simples: o repositório GitHub funciona como a “pasta oficial” do projeto. Nele ficam guardados os ficheiros, a documentação (relatório, planeamento, recursos), as evidências (prints) e um histórico das alterações (commits/atualizações). Assim, o projeto fica mais fácil de acompanhar, melhorar e entregar.

Para que serve este manual

Objetivo: apoiar o desenvolvimento do projeto do início ao fim, com organização e evidências.

Ao longo da PAP, é normal existirem várias versões de ficheiros, mudanças de ideias, correções e melhorias. Sem um método de organização, torna-se fácil perder trabalho, confundir versões ou não conseguir justificar decisões tomadas durante o desenvolvimento. Este manual ajuda a resolver isso.

Em termos práticos, vais usar o GitHub para:

  • Criar e manter um repositório do projeto.
  • Organizar pastas (código, docs, imagens/prints, recursos).
  • Registar evidências do trabalho (prints e documentos).
  • Trabalhar em segurança com branches e Pull Requests (sem estragar a versão principal).
  • Gerir tarefas com Issues (o que falta fazer, o que já foi feito).
  • Publicar um site do projeto com GitHub Pages (manual, relatório, protótipo, portefólio).
Sugestão: usa este manual como referência semanal. Sempre que avançares uma fase, volta aqui e confirma se o repositório continua limpo, organizado e atualizado.

Como usar este manual ao longo do projeto

Uma rotina simples que funciona durante toda a PAP.

1) No início

Criar base do projeto e organizar estrutura.

Arranque
  • Criar repositório (Public se vais usar Pages).
  • Definir pastas (src/, docs/, images/).
  • Adicionar um README com objetivo e plano.

2) Todas as semanas

Atualizar trabalho e registar evidências.

Rotina
  • Upload de documentos/prints importantes.
  • Atualizar o README com novidades (se necessário).
  • Usar Issues como lista de tarefas e progresso.

3) Em mudanças grandes

Trabalhar com branches e PR.

Segurança
  • Criar uma branch para alterações relevantes.
  • Abrir Pull Request para integrar na main.
  • Guardar evidências (prints do PR, merge, etc.).

4) Antes da entrega

Revisão final e publicação.

Final
  • Conferir estrutura e ficheiros finais.
  • Ativar/confirmar GitHub Pages (se aplicável).
  • Garantir links no README (repo + pages).
O que interessa é consistência: projeto organizado + evidências regulares + histórico de evolução.

Manual Interativo de GitHub para acompanhar uma PAP

Este manual foi criado como um guia de apoio contínuo ao longo do projeto de PAP. A ideia é que o GitHub funcione como a pasta oficial do projeto: organização, evidências e histórico de evolução (o que foi feito, quando foi feito e como foi melhorado).

Ao longo do trabalho, vais criar ficheiros, atualizar documentos, registar evidências e corrigir erros. Com um repositório bem estruturado, o projeto fica mais fácil de acompanhar, justificar e entregar.

Para que serve este manual

Objetivo: apoiar o desenvolvimento do projeto do início ao fim, com organização e evidências.

Um projeto bem desenvolvido precisa de organização, documentação e evidências do trabalho realizado. Este manual ajuda a manter o repositório estruturado e a criar uma rotina simples: organizar ficheiros, registar progresso, gerir tarefas e publicar resultados.

  • Organização: ficheiros e pastas no sítio certo.
  • Evidências: prints e documentos que comprovam evolução.
  • Progresso: Issues e PR como registo do que foi feito.
  • Entrega: links finais (repositório + Pages, se aplicável).
Sugestão: usa este manual ao longo do projeto (não só no início). Sempre que avançares uma fase, confirma se o repositório continua limpo, atualizado e com evidências.

Se estiveres a começar do zero: segue pela ordem do menu (Conta → Repo → Upload → Branches/PR → Issues → Pages).

Como usar este manual ao longo do projeto

Uma rotina simples, repetível e eficaz durante toda a PAP.

1) No início

Preparar a base do projeto.

Arranque
  • Criar repositório e descrição.
  • Definir pastas (src/, docs/, images/).
  • Adicionar informação inicial do projeto.

2) Todas as semanas

Atualizar e registar evidências.

Rotina
  • Upload de documentos importantes.
  • Guardar prints do progresso.
  • Atualizar Issues (tarefas e estado).

3) Mudanças grandes

Branch + PR para segurança.

Segurança
  • Criar branch para alterações relevantes.
  • Abrir PR e fazer merge quando pronto.
  • Guardar evidências (PR/merge).

4) Antes da entrega

Revisão e publicação final.

Final
  • Verificar estrutura e ficheiros finais.
  • Confirmar links do repo e Pages.
  • Garantir evidências organizadas.
